# Toche Licence Policy

Generated by M02 — Lock reuse and dependency policy.

## Scope

This policy applies to all Rust crates, npm packages, vendored data, adapted
source, fixtures, and reference materials used in Toche 1.1.0.

## Permitted licence expressions

Toche is distributed under Apache-2.0. The following licence expressions are
permitted for direct and transitive dependencies, vendored data, and adapted
source:

- Apache-2.0
- MIT
- MIT-0
- BSD-2-Clause
- BSD-3-Clause
- ISC
- Unlicense
- Zlib
- BSL-1.0
- CDLA-Permissive-2.0
- MPL-2.0
- LGPL-2.1-or-later
- Unicode-3.0
- Apache-2.0 WITH LLVM-exception
- OpenSSL
- Dual licences combining any of the above with an OR clause (e.g.
  "MIT OR Apache-2.0")

## Prohibited licences

The following are prohibited in runtime, build, or development dependencies:

- GPL-2.0, GPL-3.0
- AGPL-3.0
- SSPL-1.0
- Proprietary/closed-source licences without an explicit ADR and owner approval
- Any licence not listed in the permitted section above

## Copyleft treatment

Strong copyleft licences (GPL, AGPL) are prohibited. Weak copyleft licences
(LGPL, MPL-2.0) are permitted only if they do not require Toche to be relicensed
or distributed under terms incompatible with Apache-2.0.

## Unknown licences

Dependencies with unknown or missing licence metadata are rejected. An exception
requires:

1. Verification of the actual licence file at the pinned revision.
2. An ADR in `docs/roadmaps/v1.1.0/DECISIONS.md`.
3. Owner approval.

## Dual licences

Dual-licenced dependencies may be used under either licence expression, at
Toche's discretion. The more permissive expression is preferred. The chosen
expression must be recorded in `REUSE_MANIFEST.toml`.

## Unicode and data licences

Unicode data files and derived crates (e.g. `icu_*`, `zerovec`) are permitted
under Unicode-3.0. Vendored pricing data must be under a permissive licence
(MIT or Apache-2.0) and transformed into a compact, deterministic snapshot.

## Generated fixtures and datasets

Generated fixtures and datasets must:

- Be derived from permissively licenced sources or independently authored.
- Include provenance and licence in `REUSE_MANIFEST.toml`.
- Not introduce runtime network dependencies.
- Be deterministic and reviewable.

## GitHub Actions

GitHub Actions used in CI must be pinned by full commit SHA. Third-party actions
require an ADR. Official GitHub actions (`actions/*`) are permitted.

## npm dependencies

npm dependencies are limited to the installer (`npm/install.js`,
`npm/install.test.js`). Runtime npm dependencies are prohibited unless approved
by an ADR.

## Exception process

Any exception to this policy requires:

1. An ADR in `docs/roadmaps/v1.1.0/DECISIONS.md`.
2. Owner approval.
3. Recording in `REUSE_MANIFEST.toml` with status `conditional`.

## Attribution requirements

All approved external sources require attribution in at least one of:

- `THIRD_PARTY_NOTICES.md` (non-Cargo reuse)
- `NOTICE`
- Generated `THIRD_PARTY_DEPENDENCIES.md` (Cargo dependency licences)

## Source-pin requirements

Every approved source must have:

- Exact crate version or commit SHA.
- Verified licence at that revision.
- Primary reuse action (DEPEND, ADAPT, VENDOR, REFERENCE).
- Source files or package metadata recorded in `REUSE_MANIFEST.toml`.

## Reporting separation

- `cargo-about` output covers Cargo dependency licences.
- `THIRD_PARTY_NOTICES.md` covers adapted source, vendored data, fixtures, and
  non-Cargo reuse.

## Update procedure

Future milestones must update `REUSE_MANIFEST.toml`, `THIRD_PARTY_NOTICES.md`,
and `NOTICE` when they:

- Add a dependency.
- Adapt source.
- Vendor data.
- Import fixtures.
- Remove a source.
- Refresh a pinned revision.
